## 一、生成原则

### 1.1 核心原则

技能生成必须遵循以下原则：

| 原则 | 说明 | 约束 |
|------|------|------|
| **契约优先** | 先定义 SKILL.md，再实现脚本 | 所有技能必须先有契约 |
| **单一职责** | 每个技能只负责一个功能 | 避免大而全的技能 |
| **最小单元** | 技能应拆分到最小可执行单元 | 便于复用和组合 |
| **目录即约定** | 遵循标准目录结构 | 无需额外注册说明 |

### 1.2 禁止行为

| 禁止 | 原因 | 示例 |
|------|------|------|
| **生成复合技能** | 不符合单一职责原则 | "build-and-flash" 应拆分为两个技能 |
| **跳过 SKILL.md** | 违反契约优先原则 | 直接创建脚本 |
| **使用错误分类** | 影响技能发现和管理 | 将硬件技能放在 software 下 |
| **违反命名规范** | 影响大模型识别 | 使用大写字母或中文命名 |

## 二、生成流程

### 2.1 七步生成法

```
步骤 1：读取 .trae/SKILL.md
    │
    ▼
步骤 2：分析技能需求
    │
    ▼
步骤 3：最小单元分析
    │
    ▼
步骤 4：确定分类和路径
    │
    ▼
步骤 5：生成技能目录结构
    │
    ▼
步骤 6：更新技能注册表
    │
    ▼
步骤 7：返回生成结果
```

### 2.2 每步要求

| 步骤 | 输入 | 输出 | 关键检查 |
|------|------|------|---------|
| 1. 读取入口 | 用户请求 | 理解生成器功能 | - |
| 2. 分析需求 | 用户自然语言 | 关键词、功能意图 | 是否为技能生成请求 |
| 3. 最小单元 | 功能描述 | 技能拆分方案 | 是否需要拆分 |
| 4. 确定分类 | 技能功能 | category、path | 分类是否正确 |
| 5. 生成结构 | 分类和路径 | 目录+文件 | 目录是否标准 |
| 6. 更新注册表 | 新技能信息 | skill_registry.yaml | 元数据是否完整 |
| 7. 返回结果 | 生成信息 | 结构化报告 | 路径是否正确 |

## 三、生成验证

### 3.1 生成前检查

| 检查项 | 检查内容 | 不通过处理 |
|--------|---------|-----------|
| **需求有效性** | 用户请求是否为技能生成 | 返回提示 |
| **分类确定性** | 能否确定分类 | 返回候选分类 |
| **路径可用性** | 路径是否冲突 | 返回冲突提示 |
| **命名规范性** | 名称是否符合规范 | 返回修正建议 |

### 3.2 生成后验证

| 检查项 | 检查内容 | 失败处理 |
|--------|---------|--------|
| **目录完整性** | 是否包含必需文件 | 补全文件 |
| **SKILL.md 有效性** | 契约是否完整 | 补充字段 |
| **注册表同步** | 是否已更新注册表 | 自动更新 |
| **路径正确性** | 路径是否符合规范 | 修正路径 |

## 四、错误处理

### 4.1 常见错误

| 错误类型 | 原因 | 处理方式 |
|---------|------|---------|
| **需求不明确** | 用户描述模糊 | 询问澄清 |
| **分类不确定** | 技能功能跨类 | 列出候选分类 |
| **路径冲突** | 技能已存在 | 返回冲突信息 |
| **注册表更新失败** | 文件权限问题 | 返回错误信息 |

### 4.2 错误响应格式

```json
{
  "status": "failure",
  "failure_category": "invalid_request",
  "error_code": "GEN_001",
  "summary": "用户需求不明确",
  "suggestions": [
    "请描述技能的具体功能",
    "请说明技能的使用场景"
  ]
}
```

## 五、最佳实践

### 5.1 生成前

1. **充分理解需求**：与用户确认技能功能
2. **分析最小单元**：判断是否需要拆分
3. **确定分类**：根据功能选择分类
4. **检查冲突**：确认路径无冲突

### 5.2 生成中

1. **遵循目录结构**：严格按标准创建
2. **先契约后实现**：SKILL.md 先于脚本
3. **保持命名一致**：使用标准命名规范
4. **记录元数据**：完整记录技能信息

### 5.3 生成后

1. **验证完整性**：检查所有必需文件
2. **同步注册表**：及时更新 skill_registry.yaml
3. **返回详细信息**：提供完整生成报告
4. **给出建议**：提供下一步操作建议
